## 第一部分：执行摘要（真实版）

### 1.1 我们是谁

我们是一支由化学与计算机专业学生组成的跨学科团队，在导师指导下，围绕“化学实验条件优化”这一真实问题，开发了一套基于 SHAP 可解释人工智能技术的实验归因优化平台。

这个项目的起点不是商业概念，也不是为了追热点，而是来源于实验室里的真实困惑：

> 一组实验做得不好，问题到底出在温度、催化剂、pH，还是配比？

这个问题看起来很简单，但在真实实验中非常常见，也非常难回答。

### 1.2 我们要解决什么问题

化学实验优化长期依赖：

- 研究者经验
- 多轮试错
- 正交实验或响应面设计

这些方法并非无效，但在高维、强非线性实验条件空间下，普遍存在：

1. 试错成本高  
2. 优化周期长  
3. 关键因素难定位  
4. 结果解释不清楚  
5. 经验难以复用  

尤其是在科研训练和高校实验场景中，研究者很容易陷入“知道结果不好，但不知道该从哪里调起”的状态。

### 1.3 我们的解决方案

智析实验通过机器学习模型 + SHAP 归因算法，将实验结果分析拆解为一个更容易被研究者理解和使用的闭环：

```text
实验数据录入 → 模型训练 → 结果预测 → SHAP归因 → 优化建议 → 报告沉淀
```

这意味着平台不仅告诉用户“这组条件可能好不好”，还能够进一步告诉用户：

- 哪些条件是关键因素
- 每个条件的贡献方向如何
- 当前结果为什么会变差
- 下一轮应该先调哪几个条件

### 1.4 我们已经做到什么程度

截至目前，我们已经完成了一个可运行的 MVP 系统，具备以下能力：

- Web 化操作界面
- 实验样本录入与管理
- 多模型训练（XGBoost / Random Forest / LightGBM）
- SHAP 局部与全局解释
- 4 类 SHAP 可视化图表
- 综合评估体系
- 智能优化建议生成
- 分析报告输出
- 可视化仪表盘

这意味着我们不是停留在概念阶段，而是已经有一个可以现场演示、可以继续迭代的真实原型系统。

### 1.5 我们对自己的判断

我们不认为自己已经做成了成熟产品，也不认为当前数据规模已经足以支撑大规模商业化。但我们认为，这个方向具备三个明确价值：

1. **问题是真问题**  
2. **方法是合理方法**  
3. **系统已经具备继续落地的基础**

真实版的意思，不是把项目说小，而是把项目说实。

## 第二部分：项目起点与真实痛点

### 2.1 项目起点：来自实验室的真实问题

本项目不是从“市场调研报告”开始的，而是从实验室真实需求开始的。

在实验研究中，一个非常常见的现象是：

- 同样的体系，换了一组参数，结果差很多
- 一轮实验失败后，很难快速判断是哪个变量出了问题
- 即使找到“较好结果”，也往往无法清楚说明“为什么它更好”

我们的导师在一次交流中提出了非常典型的问题：

> 如果我们已经做了很多组实验，为什么还不能让系统告诉我们，到底是哪几个条件最关键？

这句话实际上点出了项目的根本问题：**实验室并不缺数据，缺的是把数据变成解释和决策依据的能力。**

### 2.2 我们观察到的现实情况

我们在实验室和学院范围内接触到的情况比较一致：

- 很多研究生主要依赖经验和直觉调参
- 大量实验记录没有被系统分析
- 机器学习工具使用门槛较高
- 研究者对黑箱模型存在天然不信任

这意味着即使 AI 技术已经很成熟，如果不能把结果解释清楚，也很难真正进入实验工作流。

### 2.3 为什么这个问题值得做

这个问题之所以值得做，是因为它兼具：

- **现实痛点强**：实验优化本来就耗时耗力
- **应用范围广**：高校、科研院所、企业研发都面临类似问题
- **技术切入明确**：SHAP 正好适合解决“解释”问题
- **展示效果强**：很适合做成产品原型和比赛成果

### 3.1 我们对市场的判断是保守的

真实版计划书不回避一点：我们目前还处在早期阶段，对市场的判断应以谨慎为主，不能把所有潜在客户都当作立即可转化市场。

因此，我们把市场分成三层：

1. **能理解我们价值的人**
2. **愿意试用我们产品的人**
3. **愿意为产品付费的人**

这三者不是同一个集合。

### 3.2 目标用户优先级

#### 第一优先级：高校实验室

为什么首先选择高校实验室？

- 场景最接近我们当前已有能力
- 获取反馈成本最低
- 导师资源和校内资源更容易接入
- 更适合积累案例
- 更容易形成比赛、申报、论文等成果联动

#### 第二优先级：科研团队与科研院所

科研团队往往更重视实验沉淀和分析过程，如果平台能解决解释与复盘问题，这类用户具有一定使用潜力。

#### 第三优先级：企业研发

企业市场更有付费能力，但门槛也更高，通常需要：

- 更稳定的系统
- 更强的数据安全保障
- 更明确的 ROI
- 更成熟的服务方案

所以我们的判断是：企业市场是中后期重点，而不是当前阶段的主攻方向。

### 3.3 市场规模的真实口径

如果从宏观信息化市场来讲，实验室数字化和科研信息化空间很大。但对我们这个项目来说，更重要的是细分市场中的可触达用户。

我们更认同以下现实判断：

- 早期可以真正触达的，是少量高校实验室和课题组
- 早期最重要的，不是规模销售，而是形成高质量案例
- 只有案例成立，后续收费和扩张才有基础

因此，我们不会夸大近期市场，而是把前两年的重点放在：

- 场景打磨
- 数据积累
- 案例验证
- 用户反馈

## 第五部分：技术路线与创新点（真实版）

### 5.1 为什么选择树模型 + SHAP

我们没有一开始就使用复杂深度学习模型，而是选择了更适合实验数据特征的树模型组合：

- XGBoost
- Random Forest
- LightGBM

原因很现实：

1. 实验数据通常是表格型结构化数据  
2. 样本量通常不大  
3. 变量非线性明显  
4. 研究者对解释性要求高  

在这种场景下，树模型通常比深度神经网络更务实。

### 5.2 为什么 SHAP 是关键

机器学习能提高建模能力，但如果研究者不能理解模型，就很难真正用起来。SHAP 的价值在于：

- 解释单个样本
- 分析全局变量重要性
- 给出贡献方向
- 与图表表达天然适配

对实验场景来说，这种“解释能力”几乎比“多提高 2% 精度”更重要。

### 5.3 我们的真实创新点

#### 创新点一：把 SHAP 做成实验人员可用的产品功能

这不是简单调用一个库，而是把它变成：

- 页面按钮
- 中文变量
- 图表输出
- 建议逻辑
- 报告结果

#### 创新点二：把解释结果转化为建议

大多数解释系统只到“这个变量重要”就结束了。我们继续做到了“这个变量重要，所以你下一步应该怎么调”。

#### 创新点三：把分析过程沉淀为知识

每次分析都会留下报告，这意味着平台长期可能形成实验室自己的知识资产，而不是一次性工具。

### 5.4 我们承认的局限

真实版里必须说明局限：

1. 当前规则引擎还比较基础  
2. 当前真实数据量有限  
3. 当前模型主要聚焦分类场景  
4. 当前平台仍偏原型阶段，距离正式商用还有提升空间  

但这些局限是“可迭代的问题”，不是“方向错误的问题”。

- 第二年重点不是规模化，而是找到可复制的试点模式
- 第三年以后，才有机会形成更稳定的商业模式

### 7.2 商业模式设计

#### 模式一：高校 / 科研团队服务

可以采用：

- 试点合作
- 低价订阅
- 教学合作
- 项目共建

这是最容易启动的商业化方向。

#### 模式二：企业私有化部署

对企业客户来说，真正有吸引力的是：

- 数据不出内网
- 模块可定制
- 报告规范化
- 支持内部流程

这一模式单客价值高，但进入门槛也更高。

#### 模式三：定制化服务与咨询

包括：

- 数据清洗
- 模型适配
- 场景规则优化
- 培训与辅导

这类服务更适合项目早期，既能带来收入，也能帮助我们更理解用户需求。

我们不使用过于激进的收入预测，保守判断如下：

#### 第一年

- 重点：试点与验证
- 收入：可以为 0，也可能只有少量合作收入

#### 第二年

- 目标：形成若干试点用户

真实版的重点不是证明“我们一定能很快赚很多钱”，而是证明“这个方向有合理可行的转化路径”。
