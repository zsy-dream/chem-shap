# 05 - 技术实现

## 技术架构概览

### 整体架构设计
```
 前端展示层：Jinja2 模板 + HTML/CSS/JavaScript（页面展示与交互）
 路由控制层：Flask 路由（页面路由 + REST API）
 业务服务层：DataService（数据清洗/编码）+ AttributionService（SHAP解释与图表生成）等
 数据存储层：SQLite（演示/开发）+ 模型文件存储（.pkl）
```

### 核心技术栈
- **后端框架**：Flask 2.x + Flask-SQLAlchemy + Flask-Login + Flask-CORS
- **机器学习**：scikit-learn +（XGBoost / LightGBM / RandomForest 等算法在训练阶段可选）
- **可解释AI**：SHAP（TreeExplainer / KernelExplainer）
- **数据处理**：Pandas / NumPy
- **图表生成**：Matplotlib（后端生成PNG并转Base64）
- **数据库**：SQLite（当前实现）
- **部署**：提供脚本启动与 Docker 部署方式（以仓库配置为准）

## 数据模型（数据库表）

系统使用 Flask-SQLAlchemy 定义核心对象（见 `app/models.py`），用于支撑样本、模型、报告的可追溯管理：

- **User**：用户账号、角色、创建时间
- **Sample**：样本编号、实验轮次、实验分组
- **ExperimentRecord**：单条实验记录（`feature_data` 以 JSON 形式存储）
- **MLModel**：模型元信息（名称、版本、类型、模型文件路径、评估指标、是否激活）
- **OptimizationReport**：分析报告（成功概率、SHAP值、Top特征、专家建议文本、创建时间）

## 核心流程与实现要点

### 1) 数据导入与预处理（DataService）

系统在 `app/services/data_service.py` 实现了面向表格数据的基础预处理能力：

- **多格式加载**：CSV / Excel
- **缺失值处理**：
  - 数值列使用中位数填充（median）
  - 类别列使用众数填充（most_frequent）
- **简单异常值截断**：对数值列按 `mean ± 3*std` 进行裁剪（clip）
- **编码与标准化**：
  - 类别变量使用 `LabelEncoder`
  - 数值变量可使用 `StandardScaler`

这套流程的目标是：降低用户导入数据时的“脏数据”门槛，保证训练/分析的基本可用性。

### 2) SHAP 归因解释与可视化（AttributionService）

系统在 `app/services/attribution_service.py` 封装了 SHAP 的核心能力：

- **解释器选择**：
  - 若模型具备 `predict_proba`，默认使用 `shap.TreeExplainer`
  - 否则使用 `shap.KernelExplainer`，并要求提供背景数据 `X_background`
- **SHAP值计算**：兼容二分类输出（当 `shap_values` 为 list 时取正类 `[1]`）
- **全局重要性**：对每个特征取 `mean(abs(shap))` 得到排序
- **局部解释**：输出单样本各特征的贡献，并按绝对值大小排序
- **图表生成**：
  - Summary Plot（摘要图）
  - Waterfall Plot（瀑布图）
  - Force Plot（力导向图，matplotlib渲染）
  - Dependence Plot（依赖图）
- **输出格式**：后端生成 PNG 后转为 Base64 字符串，便于直接嵌入网页模板和报告

### 3) 报告存储与“可追溯”

分析结果会写入 `OptimizationReport`：

- `success_probability`：模型预测的达标概率
- `shap_values`：特征贡献字典（JSON）
- `top_features`：关键特征列表（JSON）
- `expert_advice`：建议文本（当前实现可存储规则/模型生成的建议内容）

这保证了“每次分析都有记录”，便于复盘与对比。

## 系统模块设计

### 数据管理模块
- **功能**：样本/实验记录管理、数据导入（CSV/Excel）、基础校验
- **技术**：SQLAlchemy ORM、Pandas数据处理

- **技术**：scikit-learn 生态 + 树模型算法（以项目代码实现为准）
- **特性**：模型以文件形式持久化，并在数据库中保存元信息与指标

### 归因分析模块
- **功能**：SHAP计算、可视化生成、建议生成
- **技术**：SHAP + Matplotlib
- **特性**：图表统一转Base64，便于Web端展示

### 报告系统模块
- **功能**：报告生成、存储、查询、导出
- **技术**：数据库存储（JSON字段）+ 页面模板渲染
- **特性**：报告可追溯、可二次展示（用于科研汇报/答辩材料截图）

## 关键实现细节（与当前版本一致）

### 图表输出为什么选 Base64

当前版本采用服务端绘图（Matplotlib）并将图片转 Base64 字符串返回，优点是：

- 浏览器端无需额外图表渲染依赖
- 结果可直接存到报告中，便于复盘
- 在服务器无显示环境下也能稳定生成图（`matplotlib.use('Agg')`）

### 为什么需要“激活模型”

系统支持多模型训练与管理。通过 `MLModel.is_active` 标记当前默认使用的模型，能避免：

- 用户在多模型间切换导致分析结果不一致
- 后续复盘时无法对应当时使用的模型版本

## 部署与运行方式（与仓库一致）

当前项目支持本地脚本方式启动（如 `run.py`），也提供 Docker 相关配置用于部署（如 `docker-compose`）。在比赛/答辩场景中，推荐使用固定版本的演示数据与演示模型，保证可复现。

## 部署架构

### Docker容器化
```dockerfile
FROM python:3.8-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 5001
CMD ["python", "run.py"]
```

### Docker Compose编排
```yaml
version: '3.8'
services:
  web:
    build: .
    ports:
      - "5001:5001"
    environment:
      - FLASK_ENV=production
    volumes:
      - ./data:/app/data
      - ./models:/app/models
    # 注：当前版本以 SQLite 演示为主；如需 MySQL 可在后续迭代中加入数据库服务
```

## 合理的后续扩展方向（规划，不夸大）

- **更多目标类型**：从二分类扩展到回归（如产率/纯度等连续指标）
