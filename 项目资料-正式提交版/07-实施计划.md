# 07 - 实施计划

## 项目总体规划

### 项目愿景
把“可解释AI + 实验数据分析”做成实验室能用、能复盘、能沉淀的工具，先在高校/课题组完成试点验证，再逐步扩展到更多场景。

### 项目目标
- **短期目标（0-6个月）**：完善MVP稳定性与演示链路，完成1-3个试点案例
- **中期目标（6-12个月）**：形成可复制的试点流程（数据模板、操作手册、报告模板），扩展到更多课题组
- **长期目标（12个月以后）**：在真实合作需求驱动下，逐步增强企业级能力（权限、审计、对接、私有化部署）

## 第一阶段：产品验证期（0-6个月）

### 主要目标
- 完成MVP产品功能优化
- 在实验室环境中验证可行性
- 积累首批实验数据与案例
- 建立初步用户群体

### 具体任务

#### 技术开发（第1-2个月）
- **任务1**：完善核心算法模块
  - 统一SHAP解释输出（局部解释、全局重要性、Top特征）
  - 统一图表生成与展示（Base64输出稳定）
  - 建议模块从“可解释结果”出发，形成可落地的方向性建议模板
  - **里程碑**：可对演示数据稳定生成图表与报告，并可复盘历史结果

- **任务2**：优化用户界面与体验
  - 梳理最短演示路径（导入→训练→激活→分析→报告）
  - 优化关键页面提示与错误信息（数据缺列/类型错误/模型未激活等）
  - **里程碑**：非开发者按照说明可独立跑通一次完整流程

- **任务3**：完善系统稳定性
  - 修复已知问题，补齐异常处理（数据为空、字段缺失、SHAP解释器未初始化等）
  - 补充日志与可定位的错误提示，方便现场演示
  - **里程碑**：在固定演示环境下可连续演示多次，不出现影响展示的报错

#### 数据与场景验证（第3-4个月）
- **任务4**：收集真实实验数据
  - 与1-2个课题组/实验室沟通数据字段与记录方式
  - 形成统一的导入模板（CSV字段、单位、缺失值规则）
  - **里程碑**：拿到至少1份可用于训练与解释的真实数据（可脱敏）

- **任务5**：验证算法有效性
  - 在真实数据上跑通训练与解释，验证输出是否符合领域直觉（与导师/研究生一起评审）
  - 明确“可解释输出”在实验复盘中的作用：能否快速定位关键因素与负贡献因素
  - **里程碑**：形成1份试点评估记录（包含截图、Top特征、建议与复盘结论）

- **任务6**：形成典型案例
  - 优先完成1个“可讲清楚”的典型案例（解释逻辑清晰、图表可展示）
  - 在条件允许时扩展到2-3个案例（不同体系/不同课题方向）
  - **里程碑**：至少1个完整案例包（数据模板+截图+报告+讲解稿）

#### 用户测试与反馈（第5-6个月）
- **任务7**：用户测试
  - 邀请课题组内2-5名研究生试用
  - 收集“哪里看不懂/哪里容易误用/哪里需要提示”的反馈
  - **里程碑**：形成可执行的改进清单（按优先级）

- **任务8**：产品迭代
  - 完成一次版本迭代（以“演示稳定性+说明清晰”为优先）
  - 输出一套比赛可用材料：系统截图页、演示脚本、常见问题处理
  - **里程碑**：形成“答辩可用版本”（功能稳定、展示一致、可复现）

### 资源需求
- **人力资源**：以项目现有团队为主（前后端/算法/产品协同）
- **技术资源**：一台可演示的电脑或服务器（本地/校园网均可）
- **数据资源**：至少1份真实数据（可脱敏），用于验证与案例展示
- **资金需求**：以学校支持/竞赛支持/试点合作经费为主，优先保障演示与试点推进

### 风险控制
- **技术风险**：预留技术缓冲时间
- **数据风险**：多渠道数据来源
- **用户风险**：提前锁定测试用户

## 第二阶段：场景打磨期（6-12个月）

### 主要目标
- 增强产品功能与稳定性
- 扩大用户群体与案例
- 探索商业化模式
- 建立品牌影响力

### 具体任务

#### 产品功能扩展（第7-8个月）
- **任务9**：增加高级功能
  - 视试点需求增加回归目标支持（例如产率/纯度等连续指标）
  - 完善数据导入校验（字段、单位、范围）
  - **里程碑**：在一个真实场景中跑通“回归/分类之一”的稳定流程

- **任务10**：完善企业级功能
  - 若出现企业/院所合作需求，再逐步补齐更严格的权限与审计能力
  - **里程碑**：形成可选的私有化部署方案与交付清单（按真实需求推进）

- **任务11**：增强集成能力
  - 开发API接口
  - 支持数据导入导出
  - 集成第三方工具
  - **里程碑**：集成能力完善

#### 市场推广（第9-10个月）
- **任务12**：建立品牌影响力
  - 参加创新创业比赛
  - 形成对外可展示的案例与演示视频
  - **里程碑**：有1套可复用的路演材料（PPT + Demo + 常见问答）

- **任务13**：扩大用户群体
  - 通过导师与学院渠道拓展到更多课题组试点
  - **里程碑**：累计完成3-5个可复盘案例（可脱敏）

- **任务14**：建立合作网络
  - 与校内外课题组建立长期试点沟通机制
  - 在条件允许的情况下，与1-2位行业人士/企业工程师交流（用于需求理解与验证，不强行绑定商业合作）
  - **里程碑**：形成稳定的数据获取与需求反馈路径

#### 商业化探索（第11-12个月）
- **任务15**：试点收费
  - 以“试点合作+培训支持+定制报告模板”为主，不急于规模化收费
  - **里程碑**：形成1个可签约的试点合作模板（交付物+验收指标+支持内容）

- **任务16**：完善商业模式
  - 优化定价策略
  - 完善服务流程
  - 建立销售体系
  - **里程碑**：商业模式确立

### 资源需求
- **人力资源**：以现有团队为主，必要时引入导师/同学协助
- **市场资源**：以竞赛展示、校内宣讲、导师推荐与试点口碑为主
- **技术资源**：优先采用本地/学校资源部署；如确有需要再申请少量云资源
- **资金需求**：以学校支持、竞赛支持与试点合作经费为主（给出区间即可，不做夸张融资口径）

### 风险控制
- **市场风险**：多元化推广策略
- **竞争风险**：保持技术创新
- **资金风险**：分阶段预算与资源保障（校内支持、竞赛支持、试点合作经费为主）

## 第三阶段：扩展与落地期（12-24个月，按真实需求推进）

### 主要目标
- 在高校/科研场景形成更稳定的使用方式与案例沉淀
- 若出现明确企业合作需求，再推进私有化部署与数据安全增强

### 重点任务
- **任务17：案例库与模板化**
  - 建立“案例包”标准：数据模板、字段说明、分析截图、报告结论、复盘要点
  - **里程碑**：累计沉淀 5-10 个可复盘案例（可脱敏）

- **任务18：部署与运维规范化**
  - 输出更标准的部署说明、常见问题排查、备份与升级流程
  - **里程碑**：试点单位可按文档独立部署/升级（或在支持下完成）

- **任务19：企业合作（可选）**
  - 在真实合作需求出现后，逐步补齐：更严格的权限、审计日志、数据库切换（SQLite→MySQL）、内网部署方案
  - **里程碑**：完成 1 个企业/院所试点（以“可运行+可解释+可追溯”为验收）

- **任务22：生态合作（可选）**
  - 以“试点场景合作”为主，优先与高校/科研团队建立持续使用关系
  - 如遇到合适机会，再与仪器厂商/行业伙伴交流对接需求
  - **里程碑**：形成可复用的合作与交付模板（数据模板、报告模板、部署与培训清单）

### 资源需求（第三阶段）

- **人力资源**：以现有团队为主，按试点规模逐步增加协作人员
- **技术资源**：以本地部署/学校资源为主，必要时使用少量云资源
- **资金需求**：以试点合作经费、学校支持为主（按实际推进情况列预算）

## 本阶段关键里程碑（更贴近比赛/试点）

- **里程碑1：演示可复现**
  - 在固定环境下，能稳定完成“导入→训练→激活→SHAP解释→报告”流程
- **里程碑2：案例可讲清楚**
  - 至少1个典型案例包（数据模板+截图+结论+讲解稿），可用于答辩
- **里程碑3：试点可持续**
  - 与1-2个课题组形成持续试点反馈机制，能滚动迭代产品

## 风险管理

### 技术风险
- **风险识别**：技术路线错误、技术实现困难
- **应对措施**：技术调研、原型验证、专家咨询
- **应急预案**：备选技术方案、外部技术合作

### 市场风险
- **风险识别**：市场需求变化、竞争加剧
- **应对措施**：市场调研、用户反馈、产品迭代
- **应急预案**：调整产品定位、寻找新市场

### 管理风险
- **风险识别**：团队管理困难、资源不足
- **应对措施**：完善管理制度、资源储备
- **应急预案**：外部合作、校内资源协调支持

### 财务风险
- **风险识别**：资金不足、收入不及预期
- **应对措施**：分阶段预算、成本控制
- **应急预案**：调整商业模式、寻求投资
